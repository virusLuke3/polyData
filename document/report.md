# polyData 当前数据库分析报告

## 1. 执行摘要

截至 2026-03-11，polyData 当前数据库已经不是“样例仓库”阶段，而是一个具备真实规模的 Polymarket 链上数据仓库。当前 MySQL 库中，核心三张业务表已经累计沉淀：

- markets：606,270 条
- trades：436,395,467 条
- oracle_events：492,528 条

如果把 sync_state、block_timestamps、uma_adapter_mapping 一并纳入，当前数据库已经形成“市场定义 + 链上成交 + 预言机生命周期 + 同步状态 + 时间戳缓存 + 语义映射”的完整结构。

从数据规模上看，trades 是绝对主体，说明仓库的主要存储压力和分析价值都集中在高频成交流；markets 承担市场主索引作用；oracle_events 负责结果裁决与争议过程追踪。对团队而言，这意味着该库已经足以支撑市场活跃度监控、价格形成研究、结算质量分析，以及更细粒度的事件驱动研究。

- oracle_events 约 49.3 万条，说明不是所有 market 都已经进入完整裁决流程，但数据库已经拥有足够大的结果事件样本池。

## 3. 数据库组成与表结构

polyData 当前数据库由 6 张核心/辅助表组成，可以分成三层：

### 3.1 业务主数据层

#### markets

这张表是全库的主索引。关键字段包括：

- id：数据库内部主键
- slug：市场 URL 级别唯一标识
- condition_id：CTF 条件主键，也是链上关联的核心锚点
- question_id：问题标识
- oracle：oracle 合约地址
- yes_token_id / no_token_id：YES/NO outcome token 唯一标识
- title / description：市场文本信息
- created_at / end_date：市场时间信息
- category / tags：市场分类与标签
- clob_token_ids：CLOB token 关联信息

这张表回答的是：“这个市场是什么？”

#### trades

这张表保存每一笔链上撮合成交明细。关键字段包括：

- tx_hash / log_index：链上唯一事件定位
- market_id：所属市场
- maker / taker：成交双方
- price / size：价格和成交量
- side：BUY 或 SELL
- outcome：YES 或 NO
- token_id：具体成交的 outcome token
- block_number / timestamp：成交发生的链上时间坐标

这张表回答的是：“这个市场如何被交易？”

#### oracle_events

这张表记录市场进入 UMA/Oracle 生命周期后的关键事件。关键字段包括：

- tx_hash / log_index / block_number：链上事件定位
- event_time：事件时间
- event_status：request、propose、dispute、settle 等状态
- market_id / condition_id / question_id：与市场的关联键
- proposed_price / settled_price：提案价格与结算价格
- payout：结算赔付结果
- proposer / disputer / settlement_recipient：关键参与方

这张表回答的是：“这个市场最终如何被裁决？”

### 3.2 运行保障层

#### sync_state

记录各同步链路的断点位置，保证市场、交易、oracle 同步支持断点续传和失败恢复。

#### block_timestamps

缓存 block_number 到 timestamp 的映射，避免重复 RPC 查询，降低同步与回放成本。

#### uma_adapter_mapping

补足 oracle 文本语义与适配器映射，使链上裁决事件更容易和业务 market 对齐。

## 4. 简单数据分析

### 4.1 数据组成分析

当前数据库不是一个“市场列表库”，而是一个明显偏交易流的数据仓库：

- markets 负责定义市场对象，是低频更新的主索引层。
- trades 负责承载高频成交流，是体量最大、分析价值最高的数据层。
- oracle_events 负责结果判定，是低频但高价值的生命周期事件层。

这说明 polyData 的结构非常适合做以下三类分析：

- 横截面分析：看当前有哪些市场、属于哪些类别、是否已到期。
- 时间序列分析：看交易价格、交易量、成交密度如何随时间变化。
- 结果归因分析：把市场定义、交易行为和最终结算结果串成闭环。

- politics、weather 这类非纯金融主题也已经具备可观规模，说明后续做横向主题研究是可行的。

### 4.3 数据质量观察

基于当前快照，可以直接观察到两类值得注意的问题：

- markets.end_date 的最早值为 2011-07-05，显著早于市场创建覆盖起点 2020-10-02，说明源数据存在少量异常时间值或兼容性字段污染。
- market、trade、oracle 三张表的时间覆盖并不完全重合，说明这是一个分阶段构建的数据仓库，后续在做全样本回测时需要明确每张表的有效起点。

这类问题不影响当前数据库的整体可用性，但前端展示、指标回测和研究报告中应明确样本起止边界。

## 6. 结论

当前 polyData 数据库已经具备三个非常明确的特点：

1. 市场主表覆盖时间长，能够提供稳定的市场索引层。
2. 交易表规模已经达到 4.36 亿级，足以支持高频交易行为和价格形成研究。
3. 预言机事件表已经具备 49.2 万级生命周期记录，能够把市场结果、争议过程和交易行为真正打通。

如果后续要把这套数据库用于前端可视化或分析产品化，建议优先围绕以下几类指标建设页面：

- 市场总量、活跃市场数、已结算市场数
- 市场类别分布
- 单市场价格时间序列与成交量曲线
- 预言机 request / propose / dispute / settle 时间线
- 结算前价格与最终 settled_price 的偏离度分析

从当前这份数据库快照看，polyData 已经不只是一个数据采集工程，而是一个可以直接支撑研究、可视化和量化分析的真实数据底座。
